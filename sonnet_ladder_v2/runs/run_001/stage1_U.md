## U1-B — GGUF Tensor Offset Alignment to NVMe Sector Boundaries (NTOB)

**Track B — compression / I/O layout.** The GGUF file format stores each tensor at a byte offset recorded in a flat header. On NVMe hardware, reads that cross a 4 KB sector boundary require two sector fetches from the controller queue; reads that are sector-aligned complete in one. On Qwen3-70B at Q4_K_M (~40 GB), a naive gguf_convert-produced file places tensor starts at arbitrary offsets relative to 4096-byte sector boundaries. The substrate primitive here is the NVMe controller's internal read scheduler: a read of N bytes at offset O uses ceil((O mod 4096 + N) / 4096) sector fetches. For a 70B model where inference is NVMe-bandwidth-bound, every extra sector fetch per tensor load adds latency. The proposal: repack the GGUF file so that every weight tensor with byte count ≥ 8192 begins at a 4096-byte-aligned offset (padding the preceding tensor with zeros up to the next 4K boundary). The alignment padding is never loaded — file seeks jump over it. Total storage overhead: at most (n_tensors × 4095) bytes; for a 700-tensor model this is ≤ 2.8 MB on a 40 GB file, < 0.01%. The OS Windows Cache Manager and the NVMe firmware both schedule at sector granularity. After alignment, every tensor load that fits in N full sectors completes with no split-sector penalty. Not a CF1–CF12 derivation; the mechanism is purely substrate-geometric. Closest kill-list item is R1/S2 "OS-Paging-Aware Weight Layout" (killed for not reducing bpw). NTOB differs: it does not change bpw, but it reduces the sector-fetch count per tensor, which is the binding I/O primitive on NVMe-bottlenecked inference. The bpw argument for that kill was wrong for NVMe-bottlenecked inference; the primary risk is that the workload is NOT NVMe-bottlenecked on this hardware, which the smallest experiment settles.

**Residency arithmetic.** At Q4_K_M, Qwen3-70B is ~40 GB; at ~3 GB/s NVMe sequential throughput the raw transfer takes ~13 s/token on a 700-weight-per-token load pattern. Split-sector reads add overhead proportional to the fraction of tensor-boundary reads that are misaligned. If half of 700 per-token tensors are misaligned (empirically measurable), and each misaligned read costs one additional 4 KB sector fetch at 4 KB / 3 GB/s = 1.4 µs extra latency, total overhead is 350 × 1.4 µs = 490 µs/token (0.05% of the 13 s baseline). If the bottleneck is at queueing (NVMe QD=1 on the ik_llama.cpp path), the overhead per misaligned read can be 50–200 µs (one full round-trip per queue stall). Under QD=1, 350 misaligned reads × 100 µs = 35 ms/token, or ~3% of 1.3 s NVMe token time. The experiment measures the actual fraction and the actual per-miss penalty; the savings are substrate-measured, not assumed.

**Novelty gloss.** Closest kill-list item: R1/S2 "OS-Paging-Aware Weight Layout" — killed for not reducing bpw, which is correct but orthogonal to this proposal's claim (latency reduction via sector alignment, not bpw). Published landscape: llama.cpp GGUF tool does not enforce sector alignment; Apple LLM-in-a-flash paper addresses bandwidth but not alignment geometry; no published work applies NVMe sector-boundary alignment to LLM weight layout specifically. The structural difference is: NTOB is a zero-cost repack that exploits an existing NVMe queue scheduler primitive rather than changing model weights.

**Smallest experiment.** Testable claim: a sector-aligned GGUF of Qwen3-1.7B on the Ryzen 5 7530U NVMe delivers measurably fewer I/O stalls per token than the unaligned original, measured by Windows Performance Toolkit `xperf` disk I/O counts or by `ReadFile` call timing via a Python wrapper. Runtime: write a 20-line Python script to repack a GGUF by inserting alignment padding, run both versions under ik_llama.cpp with `--n-predict 10`, measure wall-clock and total bytes read via Sysinternals Process Monitor. Total runtime ≤ 2 hours. Go/no-go threshold: >2% wall-clock speedup per token on a model whose weights don't fit in RAM (must force NVMe path). Structural finding on NO-GO: the NVMe path in ik_llama.cpp does not trigger split-sector reads in the observable workload (already page-aligned by mmap or by Windows I/O manager buffering), which would mean the Cache Manager absorbs misalignment above us — itself a substrate finding that closes the alignment sub-question permanently.

**Primary risk.** Windows Cache Manager may round-up all reads to page-aligned boundaries internally before the NVMe controller ever sees them, making tensor-start alignment irrelevant at the controller level. Mitigation: measure raw sector I/O counts via `xperf` or ETW traces on the storage stack, not just wall-clock — if the Cache Manager is masking misalignment, the sector-count trace will be flat across aligned and unaligned versions.

## U2-B — zstd Dictionary Pre-Warm on GGUF Block Metadata (ZDPW)

**Track B — I/O / entropy.** zstd with a pre-trained dictionary achieves near-ANS entropy coding efficiency on small blocks when the dictionary captures the block's statistical structure. GGUF Q4_K_M block metadata — the per-block scale (d) and min (m) values stored as fp16 at 2 bytes per 32-weight block — has a stable statistical distribution across a model: scale values cluster tightly in [0.01, 0.3] with a heavy-tailed right skew; min values are near-symmetric around zero. A zstd dictionary trained on 10 MB of block metadata from calibration passes of Qwen3-1.7B captures this distribution in ~100 KB. At inference time, loading the block metadata from NVMe first as a 100× smaller zstd-compressed stream, decompressing in L3 cache, and then using the decompressed metadata to schedule which full weight blocks to fetch — this offloads the scale/min reads to CPU decompression (< 1 µs per block) and reduces NVMe reads for metadata by up to 4× (block headers from 2 bytes fp16 → ~0.5 bytes zstd-compressed with dictionary). The substrate primitive is `ZSTD_decompressUsingDDict` from libzstd (already a dependency in llama.cpp), invoked on a pre-built dictionary blob stored as a small sidecar file. This does NOT compress the weight nibbles themselves (already at near-entropy for Q4_K_M); it compresses only the scale/min header stratum. CF11 is not directly relevant; the proposal is purely substrate-geometric. Closest to R1/S2 "Arithmetic-Coded Weight Streams + Lazy Decode" (killed for ZipServ ASPLOS 2026 prior art and AVX2 serial decode): ZDPW differs by targeting only the metadata stratum (not the weight nibbles), using a dictionary (not a generic stream coder), and requiring no changes to the inference hot path — the dictionary decompression completes before the weight GEMV begins.

**Residency arithmetic.** In Qwen3-70B Q4_K_M at 40 GB: block count = (70e9 params) / 32 params/block = 2.2B blocks; metadata size = 2.2B × 4 bytes (d+m, fp16 each) = 8.6 GB. That is 21% of the total file. If the metadata compresses to 0.5 bytes/block with the dictionary (4× reduction), the compressed metadata is 2.2 GB. The NVMe reads for metadata alone drop from 8.6 GB/pass to 2.2 GB/pass if a full-model pass is the pattern. At 3 GB/s NVMe, that is 2.1 s saved per pass (8.6→2.2 GB). For single-token inference loading one token's worth of layers, the per-token metadata load is (8.6 GB / 80 layers) / 80 layers = ~11 MB per layer; after compression ~2.7 MB. Saving: ~8 MB × (1/3 GB/s) = 2.7 ms/layer × 80 = 216 ms/token. At 1.3 s/token baseline on NVMe, this is a 17% speedup contingent on (a) metadata being the bottleneck and (b) 4× compression ratio holding. The 4× ratio must be measured; it is not guaranteed.

**Novelty gloss.** Kill list closest: R1/S2 "Arithmetic-Coded Weight Streams + Lazy Decode" — killed for ZipServ prior art and serial decode. ZDPW is structurally different: it targets metadata only (not weight nibbles), uses zstd dictionary mode (offline training, fast decode), and does not require any changes to the weight nibble decode path. Published landscape: WaterSIC addresses weight entropy but full-stream; GGUF format spec does not provide for compressed metadata; llama.cpp has no metadata-tiered I/O today. The dictionary-keyed metadata compression on a known-distribution sidecar stratum is the frame-orthogonal move.

**Smallest experiment.** Testable claim: zstd with a calibration-derived 100 KB dictionary compresses the Q4_K_M block metadata (d+m columns extracted from a GGUF) to ≤ 0.6 bytes/block (≥ 3.2× compression) on Qwen3-1.7B. Runtime: extract metadata from GGUF (10 min Python), train zstd dict on first 50% of blocks (`zstd --train`), compress remaining 50% with dictionary, measure ratio. Total runtime ≤ 1 hour. Go/no-go: ≥ 3.2× compression ratio on held-out blocks. Structural finding on NO-GO: block metadata entropy is near-uniform despite visual clustering (the tail distribution is hard), which would bound metadata compression gains and close this sub-question. Also determines whether a dictionary of 100 KB is sufficient or requires 1 MB+.

**Primary risk.** GGUF block metadata may not compress well because the scale values (while clustered in range) have high entropy within that range due to per-layer normalization variation — the distribution has many modes rather than a single tight peak. Mitigation: measure per-layer entropy of the metadata separately; if per-layer distributions are tighter than the global distribution, use per-layer dictionaries (one per layer at negligible overhead).

## U3-A — Windows VirtualLock for Attention-Weight Residency Pinning (VLAP)

**Track A (cross-track: residency management couples to computation scheduling).** The Windows `VirtualLock` API pins virtual memory pages into the working set and prevents them from being trimmed by the working-set trimmer (WST). On the Ryzen 5 7530U with 7.28 GiB RAM, the WST aggressively trims process working sets under memory pressure. During multi-turn LLM inference, the attention weight matrices (W_Q, W_K, W_V, W_O) are accessed every token for every layer. CF11 shows W_Q and W_K have concentrated spectra (r_99/d ≈ 0.63, 0.79), which means that after a rank-512 global SVD projection (CF11's GO result at ΔNLL=+0.03), the projected attention weights are 4× smaller than the originals: 4 × (d_model × K_proj) bf16 instead of 4 × (d_model × d_model). For Qwen3-70B: attention weights per layer ≈ 4 × (8192 × 8192) × 2 bytes ≈ 537 MB/layer; after global K=512 projection ≈ 4 × (8192 × 512) × 2 = 33 MB/layer. With 80 layers, full-model projected attention = 2.6 GB. Proposal: compute the global W_Q projection (K=512) and W_K projection offline, store the projected matrices as a separate mmap file, and at process startup call `VirtualLock` on those pages to pin them permanently in RAM. The MLP weights (which exceed RAM capacity and must live on NVMe) cycle through mmap as before; the attention weights never leave RAM. The substrate primitive — `VirtualLock` on Windows — prevents WST-driven eviction without requiring a kernel modification or custom driver. At 7.28 GiB RAM with 2.6 GB attention pinned, 4.68 GB remains for OS + model context + MLP weight mmap window.

**Residency arithmetic.** Qwen3-70B Q4_K_M (~40 GB total): MLP weights dominate (~32 GB for W_gate + W_up + W_down across 80 layers); attention weights account for ~8 GB total in bf16 full-rank. After K=512 global projection (CF11 GO precondition for W_Q; W_K at K=512 gives ΔNLL=+0.29 — acceptable): projected attention = 4 matrices × 80 layers × 8192 × 512 × 2 bytes = 2.68 GB. VirtualLock this 2.68 GB into RAM. Remaining RAM: 7.28 − 2.68 − 1.0 (OS) = 3.6 GB for MLP mmap window. The MLP stream from NVMe now has full 3.6 GB RAM window to prefetch into, and the attention weights are always L3-warm or RAM-resident (no NVMe latency). Per-token attention cost: 0 NVMe reads for attention (from 8 GB → 0 GB NVMe). Per-token MLP cost: still NVMe-bound at ~32 GB / 3 GB/s = ~10.7 s if full-model pass, but with the 3.6 GB window, ~3.6/32 × 100% = 11% of MLP weights are warm in RAM at any time, reducing average NVMe reads. Expected speedup from eliminating attention NVMe reads: (8 GB attention / 40 GB total) × 100% ≈ 20% of NVMe load eliminated. At 3 GB/s, 8 GB = 2.67 s saved per pass.

**Novelty gloss.** Closest kill-list item: R1/S2 "OS-Paging-Aware Weight Layout" (killed for not reducing bpw). VLAP differs: it doesn't change bpw; it uses the VirtualLock API to prevent WST-driven eviction of a specific stratum (projected attention weights), which the killed idea did not do. Published landscape: Apple LLM-in-a-flash addresses weight streaming but on macOS with madvise equivalents; no published work uses VirtualLock to partition a two-tier weight residency (attention vs MLP) at the Windows OS level. The CF11 finding (W_Q spectrum concentration at K=512) is the empirical anchor that makes the projected attention small enough to pin.

**Smallest experiment.** Testable claim: on the Ryzen 5 7530U running Qwen3-1.7B under memory pressure, pages pinned with `VirtualLock` are not evicted during a 50-token inference run while unpinned pages of equal size are evicted at least once. Runtime: Python ctypes call to `VirtualLock` on a 512 MB array; spawn parallel process to consume memory; measure page-fault counts on pinned vs unpinned arrays via `QueryWorkingSetEx` before and after. Total runtime ≤ 2 hours. Go/no-go: pinned array has 0 page faults; unpinned array has > 0 page faults during memory pressure. Also verify VirtualLock is available at the process privilege level (non-admin on Windows 11 Home requires `SeLockMemoryPrivilege` — a process-level check, not a kernel modification). Structural finding on NO-GO: VirtualLock is privilege-gated on this configuration; fallback is `SetProcessWorkingSetSize` + `mlock` via MinGW, which provides softer hints and closes the API-availability question.

**Primary risk.** `VirtualLock` on Windows 11 Home requires `SeLockMemoryPrivilege`, which is typically not granted to standard processes. Mitigation: verify privilege availability at startup; if unavailable, fall back to `SetProcessWorkingSetSizeEx` with `QUOTA_LIMITS_HARDWS_MIN_ENABLE` as the nearest equivalent (softer guarantee, still measurably reduces WST pressure). This is a configuration check, not a code change.

## U4-B — CPU Branch-Predictor-Aligned Dispatch Table for GGUF Block Type Routing (BPAT)

**Track B — runtime / I/O hot path.** GGUF stores multiple quantization types in the same file (Q4_K, Q4_0, Q6_K, F16 — depending on which tensors were quantized at what level). At inference, the block decode loop dispatches to different decode functions per block type via a switch statement or function-pointer table. On Zen 3 (Ryzen 5 7530U), the branch predictor has a bimodal pattern detector with history depth 6–8 branches. If the GGUF file is laid out such that blocks of each type are grouped (all Q4_K blocks, then all Q6_K blocks, etc.), the branch predictor settles into a single misprediction at the type boundary rather than one misprediction per type-change throughout the decode stream. Current llama.cpp GGUF files interleave different quantization types for different tensors without branch-predictor locality consideration. The substrate primitive: Zen 3's branch predictor (a perceptron-style indirect predictor, AMD patents 2020). The proposal: reorder the GGUF tensor layout so all Q4_K blocks from all layers appear contiguously, followed by all Q6_K blocks, etc. This is a pure offline file-layout change with zero model-quality impact. The benefit is in decode throughput — fewer pipeline stalls at the dispatch point. Not connected to any CF finding; the mechanism is purely microarchitectural.

**Residency arithmetic.** Branch misprediction on Zen 3 costs ~15 cycles (AMD Zen 3 optimization guide). At 3 GHz, one misprediction = 5 ns. In a Qwen3-70B Q4_K_M inference pass, block count ≈ 2.2B. With current interleaved layout, type-change frequency depends on tensor count and sizes; a conservative estimate is 1 misprediction per 1000 blocks (mostly homogeneous within each tensor, with transitions at tensor boundaries). 2.2B blocks / 1000 = 2.2M mispredictions × 5 ns = 11 ms per pass. This is a small gain (0.1% of 13 s/token). The bigger gain comes from instruction-cache locality: the decode function for a single block type fits in 1–2 cache lines; with contiguous same-type blocks, the decode function remains hot in L1I for millions of consecutive calls without eviction. If the L1I hit rate improves by 10% on decode function calls, and decode function calls constitute 30% of inference compute, effective speedup is ~3%. At 1 tok/s baseline, this is 0.03 tok/s — marginal but zero-cost to implement.

**Novelty gloss.** No kill-list item matches. Published landscape: llama.cpp kernel optimization work focuses on SIMD tile sizes; no published work addresses branch-predictor alignment of GGUF block type dispatch. Closest published idea: "SIMD-friendly tensor layout" which is about data alignment, not control-flow alignment. The structural difference: this targets the indirect branch predictor and I-cache locality of the dispatch loop, not data-side SIMD width.

**Smallest experiment.** Testable claim: grouping all Q4_K blocks before Q6_K blocks in a GGUF reduces the branch-misprediction count in the block decode loop, measurable by `perf stat -e branch-misses` (Linux) or VTune (Windows) on Qwen3-1.7B inference. Runtime: reorder a Qwen3-1.7B GGUF with a Python script (< 1 hour), run 20-token inference on both versions with hardware PMU counters, compare branch-miss counts. Total runtime ≤ 3 hours. Go/no-go threshold: ≥ 10% reduction in branch-miss count in the decode loop. Structural finding on NO-GO: the existing GGUF per-tensor grouping already provides sufficient type homogeneity that the predictor is already well-trained, meaning the layout optimization opportunity is exhausted before we try it — a useful negative finding about the predictor's effective history depth on this workload.

**Primary risk.** The decode loop's branch pattern is dominated by within-tensor type homogeneity (all blocks in one tensor are the same type), so the cross-tensor type transitions are already infrequent and the predictor already handles them with low miss rate. The gain is real but small. Mitigation: accept that this is a secondary optimization; frame as a zero-cost quality-of-life repack rather than a primary residency or throughput lever.

## U5-B — Content-Addressable Weight Block Deduplication via BLAKE3 (CAWB)

**[FREE SWING]**

**Track B — storage / residency.** Content-addressable storage systems (Git objects, IPFS, BLAKE3-keyed blobs) store unique chunks once and reference duplicates by hash. In a Qwen3-70B Q4_K_M file, quantized weight blocks may not be unique — the quantization process maps many floating-point ranges to the same nibble pattern, especially in low-information regions. The proposal: compute BLAKE3 hashes of every 32-weight block (16 bytes each for Q4_K_M); group identical blocks; store the unique set once; store the index (one 8-byte hash pointer per block) separately. The file format becomes: [unique block store] + [index]. If a fraction f of blocks are duplicates, the block store shrinks by f and storage cost is: (1 - f) × 16 bytes/block + 8 bytes/block index = 16 - 8f bytes/block vs original 16 bytes/block. Break-even at f = 0: no savings; at f = 0.25: 14 bytes/block = 12.5% savings. The substrate primitive is BLAKE3 (available as a Rust crate and a C library, both already available on Windows). At inference time, the block store is mmap'd; decode reads the index entry, jumps to the unique block, decodes. Random access within the unique block store is cache-unfriendly, but blocks accessed repeatedly (common patterns) warm up in L3. This is a `[FREE SWING]` because it has no CF1–CF12 connection and the duplicate fraction is unmeasured — it may be zero.

**Residency arithmetic.** If f = 0.10 (10% of blocks are exact duplicates), savings = 8 × 0.10 × 2.2B blocks × 16 bytes = 2.8 GB on a 40 GB model (7% reduction). If f = 0.25, savings = 5.5 GB (13.8%). If f = 0 (worst case, all blocks unique), overhead = +8 bytes/block index = +17.6 GB, making the approach counterproductive. The deduplication fraction f is the entire falsification pivot. No published measurement of block-level duplicate rates in post-quantized LLM weights exists.

**Novelty gloss.** Closest kill-list item: R1/S4-deferred Idea I "NTFS-Reflink Codebook Deduplication" — killed for Windows 11 Home limitation (reflinks not supported on NTFS in Home edition). CAWB differs: it does not use NTFS reflinks; it implements content-addressing at the application level, storing unique blocks in a custom file format. Published landscape: GGUF dedup (R5-B "GLDF") was killed for storage-only with no residency benefit — CAWB targets residency if f > 0. The structural difference from GLDF: CAWB uses per-block hashing (not filename-level dedup) and reduces the actual bytes read from NVMe by reducing the unique block count. The free-swing nature is acknowledged: if f ≈ 0, this is dead immediately.

**Smallest experiment.** Testable claim: at least 5% of 32-weight Q4_K_M blocks in Qwen3-1.7B are exact duplicates (same 16-byte nibble + scale pattern). Runtime: extract all blocks from a GGUF with a Python script, compute BLAKE3 hashes (via `blake3` PyPI package), measure collision rate. Total runtime ≤ 30 minutes. Go/no-go: f ≥ 0.05. Structural finding on NO-GO: quantized weight blocks are nearly all unique even at Q4_K_M, closing the content-addressable path for LLM weights and confirming that quantization preserves per-block uniqueness. This finding is itself informative for any future deduplication scheme.

**Primary risk.** The quantization process (especially K-means-based Q4_K) assigns different scales d and m to each block based on local range statistics, making it nearly impossible for two blocks from different layers to have the same exact 16-byte encoding even if their float values overlap in range. Mitigation: measure first (the experiment is 30 minutes); if f = 0, abandon immediately; if f > 0, the mechanism is real.

## U6-A — NVMe Queue Depth Exploitation via Async Two-Slot Layer Prefetch (AQPT)

**Track A (cross-track: computation scheduling change coupled to NVMe queue management).** NVMe SSDs expose a hardware command queue (NCQ / NVMe submission queue) that supports queue depth (QD) up to 32–64 outstanding I/O requests. On the Ryzen 5 7530U with a PCIe 3.0 ×4 NVMe SSD, the SSD's internal scheduler achieves near-peak bandwidth when the queue depth ≥ 8; at QD=1 (which is what llama.cpp's synchronous mmap path delivers on Windows), it operates at a fraction of peak. The substrate primitive: the NVMe submission queue, accessed via `ReadFile` with `FILE_FLAG_OVERLAPPED` (async I/O on Windows) or via DirectStorage (available on Windows 11 with DirectX 12 GPU, but NOT needed here). The proposal: restructure the inference loop so that while layer L is being computed (GEMV on W_gate/W_up/W_down using weights already in RAM), the I/O subsystem is concurrently prefetching weights for layers L+1 and L+2 via two outstanding `ReadFileEx` calls (double-buffer). This brings the NVMe queue depth from QD=1 to QD=2–3 during the compute window. On a PCIe 3.0 ×4 NVMe at QD=8, peak bandwidth is ~3 GB/s; at QD=1 it may be 1–1.5 GB/s (2× degradation at low QD is standard for NVMe queuing curves). If QD=2 recovers 30% of the QD=1 → QD=8 bandwidth gap, effective NVMe throughput rises from ~1.2 GB/s to ~1.6 GB/s, a 33% improvement. This translates directly to token throughput if NVMe is the bottleneck. Unlike CF13* (NTFS extent prefetch), this does not rely on prefetcher metadata — it is direct application-level async I/O. The prefetch distance (L+1, L+2) is fixed and deterministic; no prediction needed. Not a CF1–CF12 derivation. The architecture change is to the inference loop's I/O scheduling, not to the model weights.

**Residency arithmetic.** Qwen3-70B Q4_K_M at 40 GB, NVMe-bottlenecked. With QD=1: effective bandwidth 1.2 GB/s → 40 GB / 1.2 = 33 s/pass → ~0.03 tok/s. With QD=3 (double-buffer async prefetch): if bandwidth rises to 2.0 GB/s (conservative; NVMe QD curves are well-characterized), 40 GB / 2.0 = 20 s/pass → ~0.05 tok/s. 67% throughput improvement from a pure I/O scheduling change with no model modification. For Qwen3-7B at Q4_K_M (~4 GB): at 2 GB/s, this fits in RAM once warmed; the benefit is primarily the cold-start token, not steady-state. Measurement must confirm QD=1 is the actual operating point (not QD=8+ from Windows async prefetcher already running).

**Novelty gloss.** Kill-list items: R1/S2 "Cascaded Residual Codebook + NVMe Entropy Gate" (killed for QDEC overlap), R2/S2 "NVMe Prefetch Sequencer via RoPE Fingerprints" (killed for wrong axis). AQPT differs from both: it is a pure I/O scheduling change (async double-buffer) with no model change and no prediction component. Published landscape: Apple LLM-in-a-flash uses prefetching but on macOS with a different I/O stack; no published work characterizes the NVMe queue depth behavior of llama.cpp/ik_llama.cpp on Windows specifically. The deferred idea NVPF (R3/S4 Track A: "NVMe Layer Prefetch via Static Access Fingerprint") is adjacent — AQPT differs by using OS async I/O rather than a fingerprint predictor, making the mechanism deterministic (every layer is prefetched in order).

**Smallest experiment.** Testable claim: issuing two concurrent `ReadFileEx` calls for consecutive layer weight files (or mmap ranges) on the Ryzen NVMe increases effective read bandwidth by ≥ 20% vs one sequential `ReadFile` call, measured on a file large enough to not fit in OS cache (> RAM size). Runtime: write a 30-line C or Python script that reads a 4 GB synthetic file (or the actual Qwen3-7B GGUF) with QD=1 vs QD=2 async I/O, measure wall-clock. Total runtime ≤ 2 hours. Go/no-go: QD=2 delivers ≥ 20% higher throughput than QD=1. Structural finding on NO-GO: Windows ReadFileEx / mmap path already operates at QD ≥ 4 internally (the OS prefetcher is running), so there is no QD headroom to recover at the application level — the Cache Manager is already doing what we propose. This is the single most important structural question about Windows I/O on this workload, and the experiment settles it definitively.

**Primary risk.** Windows Cache Manager's sequential read detector (`ReadAhead` heuristic) may already issue multiple outstanding I/O requests behind a sequential mmap access pattern, making the application-level double-buffer redundant. Mitigation: use `FILE_FLAG_NO_BUFFERING` (direct I/O, bypassing Cache Manager) for the benchmark to isolate application-level QD from OS-level QD; compare both to establish the current operating point.

## Convergence handles

1. **NVMe queue depth under ik_llama.cpp on Windows** — whether the actual QD is 1 or higher determines feasibility of AQPT and the magnitude of NTOB's sector-miss savings; U6 and U1 both hinge on this single measurement.
2. **Windows Cache Manager ReadAhead engagement on mmap LLM access** — the central substrate question for both VLAP and AQPT: if the Cache Manager is already issuing high-QD prefetch, application-level scheduling adds nothing; if not, two ideas open simultaneously.
3. **BLAKE3 / exact-block duplicate fraction in quantized LLM weight files** — CAWB's entire mechanism lives or dies on f; also relevant to any future content-addressed weight store design.
4. **VirtualLock privilege availability on Windows 11 Home (SeLockMemoryPrivilege)** — gates VLAP; if unavailable, closes the attention-pinning path unless a softer API is sufficient.
5. **zstd dictionary compression ratio on GGUF block metadata (d+m stratum)** — ZDPW go/no-go pivot; also determines whether per-stratum entropy coding is a viable general pattern for GGUF.
6. **CF11 global W_Q K=512 projection as the attention-compression anchor** — U3/VLAP's arithmetic relies on W_Q + W_K compressing 4× via global rank projection; any idea that composes attention compression with pinning depends on CF11's GO result holding at the 70B scale.
