## U1-B — NTFS Compressed-Cluster GGUF Variant with OS-Transparent Decompression (NCCG)

**Track B — storage / I/O.** NTFS supports per-file transparent compression (`FILE_ATTRIBUTE_COMPRESSED`, compression unit 16 × cluster = 64 KB on 4 KB clusters). The NT kernel decompresses on read via the Cache Manager using the LZ77-derived LZNT1 codec entirely in kernel space — the reading process sees uncompressed bytes with no user-mode overhead. For a GGUF weight file stored as NTFS-compressed, the kernel decompresses each 64 KB compression unit as it is paged in; effective NVMe read bytes drop by the compression ratio while effective RAM bandwidth stays the same. The key question is: does Q4_K_M weight data compress under LZNT1? Standard Q4_K_M nibble data has near-uniform entropy, so raw weight nibbles will not compress. But the GGUF file contains two distinct strata: (a) quantized weight nibbles (~95% of file bytes, near-incompressible) and (b) block scale/min metadata (~5% of file bytes, fp16 values with exploitable structure per U1 run_001 finding). NTFS compression operates at the file level — it cannot compress stratum B without also trying stratum A. However, a repackaged GGUF can split the file into two NTFS files: `weights.gguf` (nibbles only, NTFS compression OFF) and `metadata.gguf` (scales + mins only, NTFS compression ON). The metadata file compresses to ~2–3× under LZNT1 (fp16 scale values clustered in [0.01, 0.3] have moderate compressibility even with LZNT1's simple LZ77 codec). The metadata file is read first per token; its compressed size is ~1/2–1/3 of the original fp16 metadata, reducing NVMe reads for the scheduling stratum. The substrate primitive: `DeviceIoControl(FSCTL_SET_COMPRESSION)` on the metadata file; no kernel modification, standard Win32. Not CF-grounded; the mechanism is purely filesystem-level.

**Residency arithmetic.** Qwen3-70B Q4_K_M: total 40 GB; metadata stratum (d + m, 4 bytes/block × 2.2B blocks) = 8.6 GB. If LZNT1 achieves 2.5× on the metadata file (conservative for clustered fp16 data), compressed metadata = 3.4 GB. NVMe savings: 8.6 − 3.4 = 5.2 GB per pass. At 3 GB/s NVMe: 5.2 GB / 3 = 1.7 s/pass saved. Baseline: ~13 s/token at QD=1. Savings: ~13% per token from metadata stratum alone, zero model-quality impact, zero user-mode code change — only a `DeviceIoControl` call at file creation. For the 1.7B test model: metadata = 8.6 GB × (1.7/70) = 209 MB; compressed at 2.5× = 84 MB; savings = 125 MB / 3 GB/s = 42 ms — measurable.

**Novelty gloss.** Closest kill-list item: run_001 U2-B ZDPW (zstd dictionary on metadata) — killed path was application-level codec; NCCG uses the OS-native NTFS compression codec (LZNT1 in kernel space), zero user-mode overhead, no dependency on libzstd. Published landscape: Apple LLM-in-a-flash uses mmap sequential access but does not exploit macOS HFS+/APFS transparent compression; no published LLM inference work uses OS-level transparent file compression on the weight file. The structural difference from ZDPW: no user-mode decompression cost, no sidecar files, no format change — the OS decompresses transparently on page-in.

**Smallest experiment.** Testable claim: `DeviceIoControl(FSCTL_SET_COMPRESSION, COMPRESSION_FORMAT_LZNT1)` on a file containing only the fp16 block-scale columns from Qwen3-1.7B Q4_K_M achieves ≥ 2× size reduction and incurs ≤ 5 ms additional CPU overhead per 100 MB decompressed. Runtime: extract metadata stratum to a flat file (Python, 20 min); enable NTFS compression; measure on-disk size vs uncompressed; read the compressed file and time decompression via wall-clock. Total runtime ≤ 2 hours. Go/no-go: on-disk size ≤ 50% of uncompressed AND wall-clock read time within 120% of uncompressed baseline. Structural finding on NO-GO: fp16 scale values from Q4_K_M are effectively incompressible under LZNT1, meaning their entropy is uniformly high despite visual range-clustering — closes the OS-transparent-compression path for GGUF strata.

**Primary risk.** LZNT1 is a simple LZ77 variant with 64 KB windows; fp16 values near 0.05 may not produce long repeated substrings that LZ77 finds, meaning compression ratio is 1.0×–1.1×. Mitigation: test on Qwen3-1.7B metadata before committing to 70B architecture; if LZNT1 yields <1.3×, consider whether delta-encoding the sorted metadata before compression (a one-time offline transform) changes the compressibility without changing the NTFS mechanism.

## U2-B — GGUF Tensor Hot-Tier Repack via Windows Superfetch Prefetch Hint File (SPHF)

**Track B — I/O scheduling / residency.** Windows Superfetch (now SysMain service, NT kernel `PfPrefetchFile` path) preloads files into the standby page list before an application requests them. Superfetch is driven by `.pf` prefetch files stored in `%SystemRoot%\Prefetch`. A custom prefetch file can be injected that tells Superfetch to preload specific byte ranges of the GGUF weight file into standby memory before inference begins. The primitive: `CreateFile` with `FILE_FLAG_SEQUENTIAL_SCAN` populates the prefetch trace; alternatively, `PrefetchVirtualMemory` (Win32, available since Windows 8.1) can be called at inference startup to issue an async batch prefetch for a specific list of virtual address ranges into the Cache Manager's standby list before the first token. The proposal differs from standard demand-paging: it issues a single `PrefetchVirtualMemory` call with an array of (base_address, length) entries covering the first 4 layers' weight extents before the model's `generate()` loop begins, allowing the NVMe to start filling the Cache Manager while the tokenizer and KV cache allocation run. This is a pure I/O scheduling change: the bytes transferred are identical; the timing shifts the NVMe I/O earlier so the first token's layer loads overlap with per-token setup cost. Substrate primitive: `PrefetchVirtualMemory` (declared in `memoryapi.h`, linked from `kernel32.lib`) — standard Win32, no admin rights, no kernel modification.

**Residency arithmetic.** Qwen3-1.7B at Q4_K_M: ~1 GB. First 4 layers ≈ 140 MB. At 3 GB/s NVMe sequential: 140 MB = 47 ms prefetch time. Tokenizer + KV allocation for a 256-token prompt on CPU: ~20–50 ms. With `PrefetchVirtualMemory` issuing the 4-layer prefetch during tokenizer time, the first token's layer-load cost reduces by min(47 ms, 50 ms) = up to 47 ms overlap. At a 200 ms/token baseline on 1.7B NVMe-resident, this is ~23% first-token latency improvement. At 70B: first 4 layers ≈ 2 GB; tokenizer time ≈ 50 ms; 2 GB / 3 GB/s = 667 ms — no overlap (tokenizer finishes before prefetch). At 70B the win is zero unless the inference framework has a longer setup phase. The mechanism is meaningful primarily at the 1.7B–7B scale where setup time is comparable to first-few-layers I/O time.

**Novelty gloss.** Closest kill-list: run_002 U-1 (PrefetchVirtualMemory for layer-by-layer overlap). That idea targeted per-layer overlap during the compute-I/O interleave window. SPHF targets first-token cold-start latency specifically: issuing a batch prefetch for the first N layers before the inference loop begins, exploiting the tokenizer setup window. The operational window is different (startup vs steady-state). Published landscape: no published LLM inference work exploits the first-token setup window as a prefetch opportunity on Windows. Apple LLM-in-a-Flash addresses steady-state streaming, not cold-start.

**Smallest experiment.** Testable claim: `PrefetchVirtualMemory` called at process startup for the first 4 layers of Qwen3-1.7B mmap reduces the wall-clock time to first token (TTFT) by ≥ 10% vs demand-paged baseline on an NVMe-bottlenecked run. Runtime: two-line patch to llama-bench wrapper; measure TTFT over 20 runs each. Total ≤ 1.5 hours. Go/no-go: TTFT reduces by ≥ 10% (statistically stable). Structural finding on NO-GO: the ik_llama.cpp startup path already issues sequential reads that saturate NVMe before the first token, meaning there is no setup-time window to exploit — a concrete substrate characterization.

**Primary risk.** `PrefetchVirtualMemory` requires the mmap to already be established; at startup the pages are not yet mapped if using lazy mmap. Mitigation: call `PrefetchVirtualMemory` immediately after `CreateFileMapping` / `MapViewOfFile`, before model parameter deserialization, to race the prefetch against initialization work.

## U3-A — NTFS Extent Defragmentation for Sequential NVMe Read Amortization (XDFR)

**Track A (I/O scheduling / computation reorder).** NTFS stores file data in extents (variable-length contiguous disk runs). A freshly written GGUF file on a lightly fragmented disk occupies 1–5 extents; after repeated writes and deletions to the same partition, a GGUF file may occupy hundreds of extents. NVMe random-seek overhead at the firmware level is effectively zero for PCIe NVMe (no rotational seek), BUT fragmented extents cause Windows Cache Manager to issue multiple smaller `ReadFile` requests rather than one large sequential DMA transfer — each request incurs a PCIe command-queue setup overhead (~2–5 µs at QD=1) and breaks the NVMe internal prefetcher's stride detector. The substrate primitive: `DeviceIoControl(FSCTL_GET_RETRIEVAL_POINTERS)` (Win32) retrieves the extent map of a file; `IFileOperation::MoveItem` or the `defrag` API can consolidate extents. The proposal: add a one-time offline step that reads the GGUF's extent map, checks fragmentation (number of extents), and if > N_threshold calls `CreateFile` + sequential copy to a defragmented layout on a clean partition region. At inference time, the defragmented GGUF yields single-extent sequential reads that the NVMe firmware's internal prefetcher can stride-detect, enabling hardware prefetch at the controller level — a substrate primitive that application software cannot invoke directly but that the NVMe firmware triggers on large sequential transfers. Not CF-grounded. The mechanism exploits the NVMe internal prefetcher (documented in NVMe spec as "Prefetching" hint via NVM Command Set), triggered only by long sequential runs detected by the firmware.

**Residency arithmetic.** 70B at Q4_K_M = 40 GB. If fragmentation causes 200 extents average, Cache Manager issues 200 read requests / token-decode pass. Each read-request setup at QD=1: ~3 µs overhead (PCIe command frame latency). Total overhead: 200 × 3 µs = 0.6 ms/pass — negligible at 13 s/token. The real gain is from NVMe firmware prefetch activation: on Samsung 980 Pro / WD SN850X class drives, internal read-ahead activates when the read pattern is detectable as sequential (>= 128 KB contiguous reads). With 200-extent fragmentation, the median extent is 200 MB; the firmware sees sequential reads within each extent and prefetches ahead within the extent. With 1-extent layout, the firmware prefetches the full 40 GB in one stride. NVMe internal read-ahead depth can add 30–50 MB lookahead without any I/O queue commands from the host — effectively raising host-visible throughput by ~10–20% for sequential patterns longer than the firmware's lookahead window. Payoff: ~1.5–2.7 s saved at 70B scale if firmware read-ahead activates (13 s × 15% = 1.95 s).

**Novelty gloss.** No kill-list item matches. Published landscape: no LLM inference paper discusses NTFS extent fragmentation and NVMe firmware prefetch interaction. Run_001 U1 NTOB addressed sector-boundary alignment; XDFR addresses extent fragmentation and firmware stride detection — orthogonal mechanisms (one is byte-offset geometry; this is extent-count and firmware-observable access pattern). The structural difference from any prior idea: the beneficiary is the NVMe firmware's own internal prefetch engine, not the OS Cache Manager.

**Smallest experiment.** Testable claim: a single-extent Qwen3-1.7B GGUF yields ≥ 10% higher sequential NVMe read throughput than the same GGUF fragmented across 50 extents, measured by `ReadFile` wall-clock on a file that doesn't fit in RAM. Runtime: write Qwen3-1.7B GGUF twice to the SSD (once fresh = 1–5 extents; once after 5 GB of writes to the same partition to induce fragmentation); query extent count via `FSCTL_GET_RETRIEVAL_POINTERS`; measure `llama-bench` per-token time. Total runtime ≤ 3 hours. Go/no-go: ≥ 10% throughput difference between 1-extent and 50-extent versions. Structural finding on NO-GO: PCIe NVMe firmware prefetch on this hardware does not distinguish between 1-extent and 50-extent access patterns, meaning extent fragmentation is a non-factor for NVMe LLM inference on this stack — closes this sub-question.

**Primary risk.** Modern NVMe drives with DRAM cache (Samsung 980 Pro, WD SN850X) may already fully absorb fragmentation overhead by caching the request map in the drive's own RAM, making extent fragmentation invisible to throughput. Mitigation: measure on both a DRAM-cached drive (the Ryzen target's actual SSD) and a DRAM-less NVMe (which shows fragmentation effects more clearly) to characterize whether the drive class matters.

## U4-B — CPU Store-Buffer Drain Scheduling for RMSNorm Scale Writeback Elimination (SBDS)

**Track B — compute / memory bandwidth.** RMSNorm in Qwen3 computes a per-token scale vector s ∈ ℝ^d_model (scale = `1 / sqrt(mean(x²) + ε)`) and writes it to DRAM as part of the activation pipeline. On Zen 3, every write to a cache-miss address allocates a store-buffer entry and eventually causes a write-back to L3 or DRAM. For a 70B model with d_model=8192, each RMSNorm layer produces one scalar + one broadcast to 8192 channels = 8192 fp16 writes. At 80 layers: 80 × 8192 × 2 bytes = 1.3 MB of RMSNorm scale writes per token. These writes cause cache-line evictions in the L3, competing with the weight matrix read stream (the dominant I/O). The substrate primitive: `_mm_stream_ps` / `MOVNTPS` (non-temporal store, x86 NT-store hint) bypasses the L3 write-allocate policy and writes directly to DRAM via the write-combining buffer, freeing L3 bandwidth for weight reads. The proposal: identify all RMSNorm scale writeback sites in ik_llama.cpp's GEMV kernel and replace standard stores with non-temporal stores for the scale vectors and intermediate activation buffers that are not reread in L3-hot state. This is a pure software change to existing code — it exploits an existing x86 microarchitecture primitive (write-combining buffer, documented in Intel/AMD architecture manuals) that ML inference code has not historically applied to RMSNorm buffers specifically. Not CF-grounded; the mechanism is microarchitectural.

**Residency arithmetic.** At 70B on DRAM: weight read bandwidth is the bottleneck at 11.5 GB/s (DRAM BW). RMSNorm writes: 80 × 8192 × 2 bytes = 1.3 MB/token. Write-allocate without NT stores: each store-miss evicts a weight-read cache line from L3 (48 MB shared L3 on Ryzen 5000 series), potentially increasing effective weight re-read cost. With NT stores: the write-combining buffer absorbs RMSNorm writes, and L3 is reserved entirely for weight reads. L3 hit rate on weight reads improves by the fraction of L3 formerly evicted by write-allocate RMSNorm stores. If L3 is 48 MB and 1.3 MB/token of write-allocate stores are displaced by NT stores: potential L3 hit-rate improvement on the 48 MB window = 1.3/48 ≈ 2.7%. At 11.5 GB/s DRAM bandwidth, a 2.7% L3 hit rate improvement saves 11.5 × 0.027 = 0.31 GB/s effective read bandwidth — marginal on DRAM-resident. On NVMe-resident (13 s/token): the dominant bottleneck is NVMe, and L3 hit rate for weights is near zero anyway; marginal. The mechanism matters most for DRAM-bottlenecked mid-size models (3B–7B at low bpw fitting in 7.28 GiB).

**Novelty gloss.** No kill-list item matches. Published landscape: NT-store optimization is standard in HPC BLAS libraries (MKL uses it for large GEMM outputs); no published LLM inference paper applies NT-store specifically to RMSNorm scale-vector writes in the activation pipeline. The structural difference from "SIMD kernel optimization" (excluded by orientation anti-frame): this uses an existing ISA primitive to route writes around the cache hierarchy, not to accelerate the compute itself. It is a cache-partition move, not a SIMD throughput move.

**Smallest experiment.** Testable claim: replacing RMSNorm scale writeback stores with `_mm_stream_si128` in ik_llama.cpp's layernorm kernel reduces per-token wall-clock time on DRAM-resident Qwen3-1.7B by ≥ 2% (target: ≥ 0.4 ms on a ~20 ms/token baseline). Runtime: two-line code change + rebuild ik_llama.cpp; measure with `llama-bench --n-predict 50`. Total runtime ≤ 2 hours including build. Go/no-go: ≥ 2% per-token speedup on DRAM-resident inference. Structural finding on NO-GO: L3 write-allocate eviction from RMSNorm stores is negligible on this workload, confirming L3 pressure is dominated by weight read traffic alone — closes NT-store as a bandwidth lever for this inference pattern.

**Primary risk.** NT stores require the data to be truly write-only (not reread before eviction from the write-combining buffer). RMSNorm scale vectors ARE reread in the subsequent GEMV (to scale the output) within the same layer, meaning the NT-store benefit evaporates if the reread latency increases. Mitigation: apply NT stores only to the post-gate activation buffers (the SwiGLU intermediate, not the RMSNorm scale), which are strictly write-once-then-read-once per token — the activation buffer is only read by the next projection, not re-used.

## U5-B — GGUF Tensor Interleave Repack for CPU Hardware Prefetcher Stride Training (HPST)

**Track B — I/O / compute.** The Zen 3 CPU hardware prefetcher includes a stride detector that predicts the next cache-line address from the stride pattern of recent misses. For LLM GEMV on weight matrices loaded sequentially row-by-row from mmap, the stride is exactly one cache line (64 bytes) — the prefetcher should fire reliably. But the Q4_K_M block layout stores data as: [block_metadata (4 bytes)] [nibbles (16 bytes)] [block_metadata] [nibbles] ... The block-metadata and nibble-payload alternate at 20-byte intervals within each block, but the block size is 20 bytes — NOT a cache-line multiple. This means cache-line boundaries fall in the middle of blocks, and the stride detector sees a non-power-of-two stride in the data stream. On Zen 3, the stride prefetcher is most effective at strides that are powers of 2 or multiples of 64 bytes. Proposal: repack each Q4_K_M block so that all block metadata is separated from all nibble payload at the tensor level — a split-field layout: tensor layout becomes [all metadata for all blocks][all nibbles for all blocks] rather than interleaved. With this layout, reading the metadata stream is a clean 4-byte-stride sequential scan, and reading the nibble stream is a clean 16-byte-stride sequential scan, both cache-line-aligned. The hardware prefetcher sees a pure stride-1 pattern (64-byte sequential) for each stream and prefetches aggressively. The decode function reads metadata first (small, fits in L1), then the nibble stream (large, prefetcher warm). The substrate primitive: Zen 3 hardware stride prefetcher (documented in AMD Zen 3 Software Optimization Guide, section 2.4 — "L1 Data Cache: Up to 64 streams with 512-byte stride detection range"). No code change beyond a one-time GGUF repacker.

**Residency arithmetic.** 70B Q4_K_M: 2.2B blocks × 20 bytes/block = 44 GB. With interleaved layout, ~30% of cache-line loads contain cross-block boundary data that forces a partial-block decode on that cache line, triggering an additional cache-line fetch for the tail bytes. If 30% of block decodes incur one extra cache-line fetch: 2.2B × 0.3 × 64 bytes = 42 GB of extra cache traffic vs pure sequential (which has zero extra). This is speculative — the actual cross-boundary fraction depends on alignment. With split-field layout: zero cross-boundary fetches; the nibble stream is a perfectly aligned sequential scan. If the extra cache traffic reduces L3 hit rate from 5% to 3% on weight reads, the DRAM bandwidth required increases by (0.05 − 0.03) / (1 − 0.05) × 11.5 GB/s ≈ 0.24 GB/s — small but measurable. The gain from prefetcher training at 1.7B scale is more tractable to measure.

**Novelty gloss.** No kill-list item matches. Run_001 U4-B BPAT targeted branch-predictor alignment of the dispatch loop; HPST targets the data-side hardware stride prefetcher for the weight read stream — structurally different (control flow vs data access pattern). Published landscape: no published LLM inference work addresses cache-line stride alignment of quantized weight block layouts specifically for CPU stride prefetcher training. The closest is GGUF format spec discussion of alignment headers, which targets mmap-page boundaries, not cache-line stride patterns.

**Smallest experiment.** Testable claim: Qwen3-1.7B with split-field GGUF layout (metadata array then nibble array) delivers ≥ 5% higher GEMV throughput on a DRAM-resident run than the standard interleaved layout, measurable by `perf stat -e L1-dcache-prefetch-misses` (Linux) or VTune prefetch efficiency metric (Windows) on a 50-token benchmark. Runtime: Python repacker + rebuild decode function for split layout + llama-bench. Total runtime ≤ 4 hours. Go/no-go: ≥ 5% GEMV throughput improvement (not full end-to-end — isolate the GEMV hot loop via llama-bench's per-layer timing). Structural finding on NO-GO: the Zen 3 hardware prefetcher handles the Q4_K_M interleaved layout well despite non-power-of-two strides, meaning the prefetcher's stream detector adapts to irregular strides and cache-line alignment of block data is a non-factor — closes this sub-question.

**Primary risk.** The Q4_K_M decode function is compute-bound (nibble unpacking + scale multiply), not load-bound, on DRAM-resident models — so prefetcher efficiency doesn't affect throughput. Mitigation: measure L1-dcache-load-misses as a proxy for prefetch effectiveness BEFORE measuring throughput; if load misses are already <2% of total loads, the mechanism has no headroom.

## U6-A — Windows Memory-Mapped I/O Large-Page Mapping for Weight Tensors (LPMT) [FREE SWING]

**[FREE SWING] Track A (residency / TLB architecture).** Windows supports large-page mappings via `MEM_LARGE_PAGES` flag in `VirtualAlloc` or `MapViewOfFileEx` (2 MB pages on x86-64 vs standard 4 KB pages). Large pages reduce TLB pressure: the Ryzen 5 7530U's L1 TLB has 64 entries for 4 KB pages but covers only 256 KB of address space; with 2 MB large pages, 64 TLB entries cover 128 MB. For DRAM-resident LLM inference where the GEMV hot loop strides across a weight matrix of size d_model × intermediate (e.g., 2048 × 6144 × 2 bytes = 25 MB for Qwen3-1.7B W_up), a standard 4 KB TLB covers 64 × 4 KB = 256 KB — covering only 1% of W_up, causing TLB thrash on every GEMV pass. With 2 MB large pages, 64 TLB entries cover 128 MB — covering all of W_up plus W_gate with headroom, eliminating TLB misses during GEMV entirely. The substrate primitive: `MEM_LARGE_PAGES` flag in `VirtualAllocEx` or via `LoadLibraryEx` on a large-page-aligned DLL; requires `SeLockMemoryPrivilege` on Windows (same gate as VirtualLock in U3 run_001). The mechanism change: at model load time, allocate weight tensors into large-page-backed memory regions rather than the default 4 KB page heap. This is a Track A proposal because it changes how the inference computation accesses memory — TLB behavior affects the effective latency of every weight-read during GEMV, which is the dominant per-token operation. Not CF-grounded; TLB behavior is independent of model structure.

**Residency arithmetic.** Qwen3-1.7B DRAM-resident at Q4_K_M: ~1 GB. GEMV for W_gate (2048×6144 = 12.6M fp16 = 25.2 MB): with 4 KB pages, 25.2 MB / 4 KB = 6300 TLB entries needed for zero-miss GEMV; actual L1 TLB = 64 entries → TLB miss rate per row ≈ (6300 − 64) / 6300 ≈ 99% on first pass (cold GEMV). TLB miss penalty on Zen 3: ~10–15 cycles for L2 TLB hit (1024 entries, 2 MB pages have dedicated L2 TLB), ~40–50 cycles for page-table walk. With 4 KB pages and 1024-entry L2 TLB covering 4 MB: W_gate (25 MB) causes L2 TLB thrash too. With 2 MB large pages: 64 L1 TLB entries cover 128 MB — W_gate fits entirely, zero L1 TLB misses after first-pass fill. Throughput impact: on a 25 MB GEMV with ~1 TLB miss per 4 KB page and ~15-cycle miss penalty: 25 MB / 4 KB × 15 cycles / (3 GHz × 16 FMA/cycle) ≈ 25600 × 15 / (3e9 × 16) = 8 µs wasted per GEMV per layer. At 28 layers × 3 MLPs each = 84 GEMVs: 84 × 8 µs = 0.67 ms TLB overhead vs ~200 ms/token total — 0.3%. Small, but the `FREE SWING` slot acknowledges this is exploratory. On 70B (d_model=8192, W_up = 8192×28672 = 234 MB): 4 KB TLB covers 0.1% of W_up; 2 MB TLB covers 55% — significant improvement in TLB hit rate for the dominant GEMV.

**Novelty gloss.** No kill-list item matches. Published landscape: no published LLM inference paper addresses TLB pressure from weight tensor access patterns specifically, despite large-page optimization being standard in HPC DGEMM (MKL uses it internally). The structural difference from "hardware-aware tiling" (anti-frame): this changes the virtual-memory page granularity via a Win32 API flag, not the computation tile size. The free-swing acknowledgment: the gain is likely small (sub-1%) at 1.7B scale; at 70B scale with 234 MB weight matrices, TLB thrash is larger and the mechanism's payoff grows with model scale.

**Smallest experiment.** Testable claim: `VirtualAlloc` with `MEM_LARGE_PAGES` for Qwen3-1.7B weight tensors reduces L1 DTLB miss count per token by ≥ 50% vs 4 KB page baseline, measurable by PMU counter `perf stat -e dTLB-load-misses` (Linux) or AMD uProf TLB miss event (Windows). Runtime: patch llama.cpp/ik_llama.cpp allocator to use `MEM_LARGE_PAGES` flag; check `SeLockMemoryPrivilege` availability; measure PMU counters on 50-token inference. Total runtime ≤ 3 hours. Go/no-go: ≥ 50% reduction in dTLB-load-misses per token. Structural finding on NO-GO: Ryzen 5 7530U L2 TLB (1024 entries × 4 KB = 4 MB coverage) is sufficient to warm all hot weight regions for 1.7B inference, making large pages redundant at this scale — motivates a 70B retest where matrices are larger than L2 TLB coverage.

**Primary risk.** `MEM_LARGE_PAGES` requires `SeLockMemoryPrivilege` on Windows (same gate as VirtualLock). On Windows 11 Home without admin rights, large pages are unavailable. Mitigation: verify privilege at startup; if unavailable, document as a Windows Home limitation and test on Windows 11 Pro / Windows Server where the privilege can be granted to standard processes. The structural finding (TLB miss rate) is still measurable without large pages if a Linux boot is available on the same hardware.

## Convergence handles

1. **NVMe firmware internal prefetch activation threshold** — XDFR (U3) depends on extent count triggering the NVMe internal stride detector; this is a distinct measurement from Cache Manager QD (prior runs) — needs characterization on the actual drive model.
2. **fp16 block-scale compressibility under LZNT1 / LZ77** — NCCG (U1) pivot; if near-uniform entropy, OS-transparent compression is closed for all GGUF metadata strata.
3. **`SeLockMemoryPrivilege` availability on Windows 11 Home at standard process privilege** — gates both LPMT (U6) and any future VirtualLock / large-page ideas; one API check closes or opens this class.
4. **Q4_K_M block-stride pattern vs CPU stride prefetcher effective stride range** — HPST (U5) go/no-go; also determines whether any future block-layout repack has hardware prefetch leverage.
5. **First-token setup window duration in ik_llama.cpp (tokenizer + KV alloc time)** — SPHF (U2) mechanism only works if setup window ≥ NVMe transfer time for first N layers; this single timing measurement gates the startup-overlap class of ideas.
6. **L1 DTLB miss rate during GEMV on 1.7B and 70B weight tensors** — LPMT (U6) free-swing anchor; if TLB misses are already negligible at 1.7B, retest at 70B is the natural next rung.
